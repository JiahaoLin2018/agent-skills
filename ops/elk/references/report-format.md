# 查询报告格式规范

query action 在获取日志后，按以下三个阶段依次输出完整报告。

---

## 阶段三：标准汇报格式

**每次输出结果前，必须先输出以下完整汇报头，再展示日志明细和分析。**

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  ELK 日志分析报告
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

【查询信息】
  项目     : my-service（我的服务）
  环境     : 生产环境（prod）
  索引     : app-prod-*
  时间范围 : 2024-01-15 15:00 — 17:00  CST
  查询条件 : message 包含 "timeout"

【统计概览】
  ES 匹配总数 :  42 条
  本次展示    :  20 条（按时间升序）
  时间跨度    :  15:03:22 — 16:58:41
  涉及主机    :  2 个（192.0.2.1 / 192.0.2.2）
  唯一 traceId:  8 个

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**字段说明**：
- `项目` — 匹配到的项目 key + description（未指定项目时显示「未指定（使用默认索引）」）
- `环境` — `生产环境（prod）` 或 `测试环境（uat）`
- `索引` — 实际查询的索引模式
- `时间范围` — 查询的时间窗口，CST 格式
- `查询条件` — 自然语言描述查询条件（而非 DSL）
- `涉及主机` — 从返回结果中统计 `ip` 或 `host.name` 的去重数量
- `唯一 traceId` — 结果中不同 traceId 的数量（无 traceId 字段时省略此行）

---

## 阶段四：日志明细展示

汇报头之后输出日志明细：

```
【日志明细】

[1]  2024-01-15 15:03:22.145  192.0.2.1
     Logger  : com.example.service.NotifyService
     Thread  : http-nio-7683-exec-3
     TraceId : abc123def456
     Codeline: NotifyService.java:87
     Message : Failed to call external API: Connection timeout after 5000ms

[2]  2024-01-15 15:03:25.891  192.0.2.1
     ...
```

**展示规则**：
- `@timestamp` 转换为 CST（+08:00）显示，格式 `yyyy-MM-dd HH:mm:ss.SSS`
- `ip` 或 `host.name` 展示在时间旁
- 空字段不展示对应行
- **消息内容**按优先级取第一个非空值：`msg` → `message` → `rest`（字段名以 2.0 探测结果为准）
- 超过 50 条时：展示前 50 条，并注明「仅展示前 50 条，如需完整数据请告知」
- 结果中发现 traceId 时，在明细末尾注明：

```
  发现 8 个不同的 traceId。是否追踪某个 traceId 的完整链路？
  （直接回复 traceId 值，如：abc123def456）
```

---

## 阶段五：分析报告

日志明细之后，**自动输出分析**：

```
【分析报告】

▌ 问题摘要
  <1-2句话概括核心问题>

▌ 根因分析
  1. 直接原因: ...
  2. 可能原因:
     - ...
     - ...

▌ 影响范围
  - 时间段  : 15:03 — 16:58（约 115 分钟）
  - 涉及实例: 2 个（192.0.2.1, 192.0.2.2）
  - 请求量  : 约 42 个（8 个唯一 traceId）

▌ 关键时间线
  15:03:22  首次出现 timeout 错误
  15:05:11  重试机制触发（Retry attempt 1/3）
  15:47:03  错误频率显著上升
  16:58:41  最后一条错误日志

▌ 建议处理
  [ 立即 ] 1. ...
  [ 短期 ] 2. ...
  [ 长期 ] 3. ...
```

**分析维度**（全部适用时纳入，信息不足时跳过对应部分）：
- 时间分布：错误是突发还是持续？有无规律？
- 错误分类：超时 / NPE / 业务异常 / 系统异常？
- 代码位置：`logger` + `codeline` 定位问题类
- 多机对比：多个 IP 同时受影响 → 全局问题；单个 → 单实例问题
- 请求关联：traceId 相同 → 同一请求链路的多个阶段
- 频率变化：与正常时段对比，是否异常增长
